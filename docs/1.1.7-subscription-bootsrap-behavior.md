# 1.1.7 Subscription / Bootstrap Behavior

## Purpose

Define deterministic post-connect sequence so each provider performs required auth, subscriptions, snapshot bootstrap, and stream start consistently.

---

## Standard Post-Connect Sequence (applies to all providers)

1. **Transport connect**
2. **Authenticate**
3. **Subscribe** (WS topics/channels where applicable)
4. **Initial snapshot request** (if supported)
5. **Begin streaming events**
6. **Mark provider ready**

If any required step fails, treat connect as failed and enter reconnect policy.

---

## Provider A (WebSocket) Checklist

- [ ] Open WS socket
- [ ] Perform auth handshake (API key/token/signature)
- [ ] Wait for auth ack
- [ ] Subscribe to configured channels/topics
- [ ] Wait for subscribe ack(s)
- [ ] Request initial snapshot (if supported)
- [ ] Validate snapshot schema/version
- [ ] Emit snapshot event(s) into dispatcher
- [ ] Start normal stream receive loop
- [ ] Mark ready/connected

Failure handling:
- Auth reject -> `AuthError` (non-aggressive retry policy)
- Subscribe reject -> reconnectable unless explicit auth/permission failure
- Snapshot timeout -> configurable: continue degraded OR fail connect

---

## Provider B (HTTP Polling) Checklist

- [ ] Validate credentials (preflight request if needed)
- [ ] Build polling cursor/window
- [ ] Request initial snapshot/baseline dataset
- [ ] Persist cursor/checkpoint
- [ ] Start poll loop for incremental updates
- [ ] Emit events to dispatcher

Failure handling:
- 401/403 -> `AuthError`
- 429 -> `RateLimitError(retry_after_seconds=...)`
- network timeout -> `NetworkError`

---

## Readiness Rule

Provider is considered **connected/ready** only after required bootstrap steps succeed:
- WS: auth + subscribe (+ snapshot if required)
- HTTP: preflight/baseline + cursor initialized

---

## Observability (structured logs)

- `post_connect_auth_start` / `post_connect_auth_ok` / `post_connect_auth_fail`
- `post_connect_subscribe_start` / `post_connect_subscribe_ok` / `post_connect_subscribe_fail`
- `post_connect_snapshot_start` / `post_connect_snapshot_ok` / `post_connect_snapshot_fail`
- `post_connect_stream_start`
- `post_connect_ready`

Include `provider`, `channels_count`, `snapshot_supported`, `duration_ms`, `reason`.
