# 1.1.11 Failure Scenarios Test Plan (Pre-Prod)

## Goal

Validate ingestion reliability under failure, not just happy path.

## Preconditions

- Supervisor + provider running in test/staging
- Structured logs enabled
- Health endpoint and metrics snapshot accessible
- Test harness can inject provider/network faults

---

## Case 1: Provider endpoint unreachable

**Inject**
- DNS blackhole or invalid host/port for provider endpoint.

**Expected behavior**
- `connect_fail` logged with `error_type=NetworkError` (or equivalent)
- state transitions: `connecting -> reconnecting`
- exponential backoff with jitter applied
- failover attempted if alternate provider enabled
- no process crash

**Pass criteria**
- retries happen with increasing delay up to cap
- supervisor remains alive
- health reports `reconnecting`

---

## Case 2: API key invalid

**Inject**
- use invalid/expired credentials (401/403 auth reject).

**Expected behavior**
- `connect_fail` logged with `error_type=AuthError`
- decision matrix marks as non-aggressive / non-retry loop
- state transitions to `stopped` (or controlled hold state)
- clear operator-visible error reason

**Pass criteria**
- no rapid retry hammering
- health reflects non-healthy terminal/auth state
- requires credential fix to recover

---

## Case 3: Network drop mid-stream

**Inject**
- start healthy stream, then cut socket/network.

**Expected behavior**
- read loop fails with network exception
- supervisor transitions to `reconnecting`
- reconnect scheduled using backoff
- after reconnect success, stream resumes
- reconnect attempt counter can reset after stable window (5 min)

**Pass criteria**
- stream recovers without restart
- message flow resumes
- reconnect_count increments, then stable reset eventually occurs

---

## Case 4: No pong responses

**Inject**
- suppress pong replies while keeping socket open.

**Expected behavior**
- heartbeat misses logged (`heartbeat_missed`)
- after K missed pongs, raise `HeartbeatDeadError` (NetworkError class)
- reconnect/failover policy triggered

**Pass criteria**
- dead connection detected within expected N/M/K bounds
- reconnect triggered automatically
- heartbeat miss metric increments

---

## Case 5: Malformed payload burst

**Inject**
- send burst of invalid/malformed messages (schema/type mismatches).

**Expected behavior**
- malformed messages marked invalid/dropped
- invalid counters increase
- system does not crash
- valid subsequent messages still processed

**Pass criteria**
- ingestion continues running
- invalid_messages_total increases as expected
- no uncontrolled memory growth / queue lockup

---

## Case 6: Rate limit response

**Inject**
- provider returns throttle signal / 429 with optional Retry-After.

**Expected behavior**
- provider raises `RateLimitError(retry_after_seconds=...)`
- supervisor schedules reconnect using Retry-After when present
- if no Retry-After, fallback to exponential backoff
- no tight retry loops

**Pass criteria**
- observed delay respects Retry-After
- reconnect attempts controlled and capped by policy
- service remains responsive

---

## Exit Criteria (Prod Readiness)

Proceed only if all cases pass and:
- no crash loops
- no retry hammering
- health/metrics/logs are consistent with expected outcomes
- recovery behavior is deterministic
