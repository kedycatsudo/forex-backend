# 1.1.4 Connection Lifecycle Workflow

## Purpose

Define an unambiguous ingestion connection state machine so runtime behavior is deterministic during startup, auth errors, timeouts, reconnects, and stop events.

---

## States

- `idle`  
  Supervisor exists but has not started connection workflow.

- `connecting`  
  Attempting to establish provider session (network connect + auth/handshake).

- `connected`  
  Session established and message stream healthy.

- `degraded`  
  Session still up, but quality is impaired (delays, repeated timeouts, missed pongs threshold not yet fatal).

- `reconnecting`  
  Previous connection considered unhealthy/failed; recovery in progress (retry/failover/backoff).

- `stopped`  
  Lifecycle ended intentionally or due to unrecoverable startup/config condition.

---

## Events / Triggers

- `startup`
- `connect_ok`
- `auth_failure`
- `timeout`
- `pong_missed`
- `pong_recovered`
- `manual_stop`
- `fatal_config_error`
- `retry_timer_elapsed`
- `reconnect_exhausted` (optional if max retries configured)

---

## State Transition Table

| Current State | Event | Condition / Guard | Next State | Required Actions |
|---|---|---|---|---|
| `idle` | `startup` | config valid + provider available | `connecting` | load settings, select provider, emit `connect_start` |
| `idle` | `startup` | fatal config invalid | `stopped` | emit `fatal_config_error`, do not start loop |
| `connecting` | `connect_ok` | auth + handshake succeed | `connected` | emit `connect_ok`, reset transient error counters |
| `connecting` | `auth_failure` | credentials rejected/unauthorized | `stopped` | emit `connect_fail` + reason=auth, mark unrecoverable |
| `connecting` | `timeout` | connect attempt timed out | `reconnecting` | increment reconnect_count, schedule retry |
| `connecting` | `fatal_config_error` | config becomes invalid at runtime init | `stopped` | emit fatal error, abort |
| `connecting` | `manual_stop` | always | `stopped` | cancel tasks, close provider |
| `connected` | `timeout` | read timeout threshold crossed | `degraded` | emit degradation signal, keep session if still open |
| `connected` | `pong_missed` | heartbeat miss count >= warn threshold | `degraded` | emit degradation signal, start recovery checks |
| `connected` | `manual_stop` | always | `stopped` | graceful close |
| `connected` | `auth_failure` | token expired and refresh failed | `reconnecting` | close session, schedule reconnect |
| `degraded` | `pong_recovered` | heartbeat restored within window | `connected` | emit recovery signal, clear degrade counters |
| `degraded` | `timeout` | repeated timeout exceeds fatal threshold | `reconnecting` | close/recycle connection, schedule retry |
| `degraded` | `pong_missed` | exceeds fatal heartbeat threshold | `reconnecting` | force reconnect |
| `degraded` | `manual_stop` | always | `stopped` | graceful close |
| `reconnecting` | `retry_timer_elapsed` | retry budget available | `connecting` | emit `connect_start`, attempt connect |
| `reconnecting` | `retry_timer_elapsed` | failover eligible | `connecting` | switch provider, increment failover_count, connect |
| `reconnecting` | `reconnect_exhausted` | max retry reached (if configured) | `stopped` | emit terminal failure |
| `reconnecting` | `manual_stop` | always | `stopped` | cancel retry loop + close resources |
| `*` | `fatal_config_error` | unrecoverable misconfiguration | `stopped` | terminate runtime |

---

## Event Semantics (to avoid ambiguity)

- **auth_failure**: explicit provider auth rejection (401/403, invalid token, handshake denied).  
  Default behavior: unrecoverable during startup (`connecting -> stopped`), recoverable during long-run only if token refresh strategy exists; otherwise reconnect then stop on repeat.

- **timeout**: no expected network/read activity within configured timeout window.

- **pong_missed**: heartbeat pong absent after ping interval; two-level handling recommended:
  - warn threshold => `degraded`
  - fatal threshold => `reconnecting`

- **manual_stop**: operator/system stop request; must preempt all other transitions.

- **fatal_config_error**: invalid required settings (missing endpoint/token/provider enabled), schema mismatch, or impossible runtime constraints.

---

## Default Recovery Policy

1. On first transport instability (`timeout`, `pong_missed`): move to `degraded`.
2. If instability persists past threshold: move to `reconnecting`.
3. In `reconnecting`, apply backoff and optional provider failover.
4. Return to `connected` only after successful connect + healthy stream.
5. Stop permanently on unrecoverable conditions (fatal config, exhausted retries if configured, startup auth failure without refresh path).

## Required Counters / Observability

- `reconnect_count`
- `failover_count`
- `last_message_at`
- current `state`
- active `provider`

Recommended structured logs:
- `connect_start`
- `connect_ok`
- `connect_fail`
- `state_transition from=<x> to=<y> reason=<event>`
- `reconnect_scheduled delay_seconds=<n>`
- `reconnect_success`
- `manual_stop`
- `fatal_config_error`

---

## Implementation Note

Centralize transitions through one helper (`_transition(to_state, reason, **meta)`) so logs/metrics are consistent and every state change is explicit.
