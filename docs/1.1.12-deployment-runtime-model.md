# 1.1.12 Deployment / Runtime Model

## Decision

Run ingestion as a **separate worker process**, not as an in-process API background task.

Rationale:
- independent lifecycle from API restarts/deploys
- clearer resource isolation (CPU/memory/network)
- safer restart behavior for long-lived websocket sessions
- better observability and scaling control

---

## Runtime Topology

- `api` service: serves HTTP endpoints only
- `ingestion-worker` service: runs supervisor loop continuously

---

## Startup Commands

### API
`uvicorn app.main:app --host 0.0.0.0 --port 8000`

### Ingestion Worker
`python -m app.ingestion.worker`

(Worker module should call `IngestionSupervisor.start()` and block until termination signal.)

---

## Restart Policy

Use Docker restart policy:

- `restart: unless-stopped`

This ensures worker auto-recovers after crashes/node reboot unless manually stopped.

---

## Graceful Shutdown

Target graceful shutdown timeout: **30s**.

Shutdown sequence:
1. receive SIGTERM
2. stop accepting new work
3. call `await supervisor.stop()`
4. close provider sockets
5. flush in-flight dispatch
6. exit cleanly before timeout

If timeout exceeded, force terminate and rely on restart policy.

---

## Health/Readiness

- API and ingestion worker expose independent health signals.
- Worker readiness = provider connected or controlled reconnecting state.
- Worker liveness = process alive + event loop responsive.

---

## Anti-Pattern (deferred)

Background task inside API app is acceptable only for short-lived local/dev setups.
Not recommended for production 24/7 ingestion.
